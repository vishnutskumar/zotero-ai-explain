# Addon

Zotero extension assets and browser-facing files. Everything under this directory ships inside
`zotero-ai-explain.xpi`.

## Structure

```text
addon/
  manifest.json          # Zotero/WebExtension manifest
  bootstrap.js           # Zotero bootstrap entrypoint; loads the IIFE bundle, threads rootURI
  content/
    zotero-ai-explain.js # esbuild IIFE bundle output (built by `npm run build`)
  llm-proxy/             # Bundled at build time from scripts/llm-proxy (excluded README.md)
    server.mjs, server.d.mts
    backends/
      codex.mjs, codex.d.mts
      claude.mjs, claude.d.mts
      ollama.mjs, ollama.d.mts
```

## How `llm-proxy/` gets here

`scripts/package-xpi.mjs` stages `scripts/llm-proxy/` into `addon/llm-proxy/` (excluding the
`README.md`) before zipping, then removes the staged copy in a `finally` block. **Do not commit
`addon/llm-proxy/`** — it is generated at build time. The plugin resolves the bundled server script
at runtime by decoding `data.rootURI` (the `file://` URL Zotero hands to `bootstrap.js`) into an
absolute path and appending `llm-proxy/server.mjs`.

## Extending

1. Keep generated build output (`content/zotero-ai-explain.js`, `llm-proxy/`) out of git unless the
   build process explicitly owns it. The `content/zotero-ai-explain.js` artifact is tracked so
   contributors don't need a fresh `npm run build` to verify the install; `llm-proxy/` is generated
   only during XPI packaging.
2. Keep browser-facing UI assets separate from provider, indexing, and conversation logic in `src/`.
3. Update the manifest when plugin permissions, entry points, or metadata change. Bump the version
   in both `manifest.json` and `package.json` together.
4. `bootstrap.js` must pass `data.rootURI` through to the IIFE bundle's `startup()` call so the
   plugin can find the bundled `llm-proxy/server.mjs`.
